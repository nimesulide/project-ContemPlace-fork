# Data Model: Multi-Tenant MVP

**Feature**: 002-multi-tenant-mvp
**Date**: 2026-03-29

## Schema Changes

### New Tables

#### `user_profiles`

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| user_id | uuid | PK, FK → auth.users(id) ON DELETE CASCADE | Same as auth.users.id |
| display_name | text | nullable | From OAuth `raw_user_meta_data` or email prefix |
| mcp_api_key_hash | text | nullable | SHA-256 hash of `cp_<hex>` key |
| plan | text | NOT NULL DEFAULT 'free' | Subscription tier |
| created_at | timestamptz | NOT NULL DEFAULT now() | |

**RLS**: `user_id = auth.uid()` for SELECT; no direct INSERT/UPDATE from client (provisioned by trigger).

#### `telegram_connections`

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| id | uuid | PK DEFAULT gen_random_uuid() | |
| user_id | uuid | NOT NULL FK → auth.users(id) ON DELETE CASCADE | |
| chat_id | bigint | NOT NULL UNIQUE | Telegram chat ID — one chat maps to exactly one user |
| connected_at | timestamptz | NOT NULL DEFAULT now() | |

**RLS**: `user_id = auth.uid()` for SELECT/DELETE.

#### `telegram_link_tokens`

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| token | text | PK | One-time token for Telegram deep link |
| user_id | uuid | NOT NULL FK → auth.users(id) ON DELETE CASCADE | |
| created_at | timestamptz | NOT NULL DEFAULT now() | |
| expires_at | timestamptz | NOT NULL | DEFAULT now() + interval '15 minutes' |

**Cleanup**: Expired tokens pruned by gardener or periodic function. Generating a new token for a user deletes all previous tokens for that user before inserting the new one.

### RPC Function Changes

#### `match_notes` — add parameter

```
p_user_id uuid  (new, required)
```

Add `AND n.user_id = p_user_id` to the WHERE clause.

#### `find_similar_pairs` — add parameter

```
p_user_id uuid  (new, required)
```

Add `WHERE a.user_id = p_user_id AND b.user_id = p_user_id` to the self-join. This ensures the cosine similarity search is scoped to a single user's notes.

### RLS Policies

Replace all `deny all` policies with user-scoped policies:

```
FOR ALL tables with user_id:
  SELECT: user_id = auth.uid()
  INSERT: user_id = auth.uid()
  UPDATE: user_id = auth.uid()
  DELETE: user_id = auth.uid()
```

Service role key bypasses these entirely. Defense-in-depth only.

### Provisioning Trigger

```
AFTER INSERT ON auth.users → public.handle_new_user()
```

Actions:
1. INSERT INTO `user_profiles` (user_id, display_name) ON CONFLICT (user_id) DO NOTHING — extract display_name from `NEW.raw_user_meta_data->>'full_name'`.
2. INSERT INTO `capture_profiles` (user_id, name, capture_voice) ON CONFLICT (user_id, name) DO NOTHING — seed with system default voice.

Both inserts use `ON CONFLICT DO NOTHING` so the trigger is safely re-runnable if a partial failure occurs and must be retried manually.

MCP API key is NOT generated by the trigger (requires returning raw key to user). Generated on first settings page visit via a dedicated endpoint.

### Migration Strategy

1. Create `user_profiles` and `telegram_connections` tables.
2. Create a "founder" user in `auth.users` (or use existing Supabase Auth user).
3. Add `user_id` column to all 7 tables as `uuid NULL` (temporarily nullable).
4. Backfill all existing rows with the founder user's ID.
5. Set `user_id` to `NOT NULL`.
6. Add FK constraints and indexes.
7. Update RPC functions with `p_user_id` parameter.
8. Replace RLS policies.
9. Create provisioning trigger.

**Order matters**: Steps 3-5 must happen in one migration to avoid downtime. Steps 7-8 can be separate migrations.

## Entity Relationships

```
auth.users (1) ──── (1) user_profiles
     │
     ├──── (many) notes
     │        ├──── (many) links (via from_id/to_id + user_id)
     │        └──── (many) enrichment_log
     │
     ├──── (many) clusters
     ├──── (many) capture_profiles
     ├──── (many) entity_dictionary
     ├──── (many) processed_updates
     │
     ├──── (0..1) telegram_connections
     └──── (many) telegram_link_tokens (ephemeral)
```

## Validation Rules

- `user_id` must reference a valid `auth.users` row (enforced by FK).
- `telegram_connections.chat_id` is UNIQUE — one Telegram chat maps to exactly one user.
- `user_profiles.mcp_api_key_hash` is nullable (key not yet generated) but must be unique when set.
- `capture_profiles` UNIQUE constraint changes from `(name)` to `(user_id, name)`.
- `entity_dictionary` UNIQUE constraint changes from `(name, type)` to `(user_id, name, type)`.
