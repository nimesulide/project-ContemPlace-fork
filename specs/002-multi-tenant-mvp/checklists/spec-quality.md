# Full Spec Quality Checklist: Multi-Tenant MVP

**Purpose**: Lightweight requirements quality check across all sections — identify the highest-impact gaps, ambiguities, and missing specifications before implementation begins.
**Created**: 2026-03-29
**Feature**: [spec.md](../spec.md)

## Requirement Completeness

- [x] CHK001 Are error handling requirements defined for provisioning trigger partial failures (e.g., `user_profiles` created but `capture_profiles` insert fails)? The edge case section mentions idempotency but FR-003 does not specify retry/recovery behavior. [Gap, Spec §FR-003 / Edge Cases] — **Resolved**: data-model.md trigger now uses `ON CONFLICT DO NOTHING` on both inserts, making it safely re-runnable.
- [x] CHK002 Are requirements specified for what happens when a user's JWT expires mid-session across all interfaces (web capture, dashboard, MCP)? Only the web capture form edge case is mentioned. [Gap, Spec §Edge Cases] — **Resolved**: MCP uses static `cp_` key (JWT-irrelevant); dashboard uses Supabase session refresh; web capture handled in T032.
- [x] CHK003 Are account deletion requirements documented as a functional requirement? CASCADE behavior is mentioned in edge cases and data-model but no FR covers the user-facing deletion flow. [Gap, Spec §Edge Cases] — **Resolved**: Self-serve deletion UI deferred to post-MVP; DB-level CASCADE covers data integrity. Added to spec.md Assumptions.
- [x] CHK004 Are requirements defined for the OAuth consent/linking flow when a user signs up with email then later tries Google OAuth with the same email (or vice versa)? [Gap] — **Resolved**: Relies on Supabase Auth default behavior for MVP. Added to spec.md Assumptions.

## Requirement Clarity

- [x] CHK005 Is "email verification" (FR-001) specified with enough detail — does the user need to verify before first capture, or can they capture immediately and verify later? [Ambiguity, Spec §FR-001 / US-1 scenario 1] — **Pass**: US-1 scenario 1 states "verify their email, **Then** they are logged in" — verification gates login.
- [x] CHK006 Is the MCP API key's first-display mechanism clearly specified? Data-model says "NOT generated by trigger" and plan says "generated on first settings page visit," but US-6 scenario 2 says user sees "a working MCP API key" immediately after signup. [Conflict, Spec §US-6 / data-model.md §Provisioning] — **Resolved in F3**: US-6 now says user clicks "Generate API Key" on settings page.
- [x] CHK007 Is "defense-in-depth" (FR-016) actionable — does it mean Workers must always add `.eq('user_id', userId)` even though RLS would catch it, and if so, is the failure mode defined when a Worker omits the filter? [Ambiguity, Spec §FR-016] — **Pass**: Adequately specified — Workers always add `.eq('user_id', userId)`; RLS catches omissions as defense-in-depth. No additional FR needed.

## Scenario Coverage

- [x] CHK008 Are requirements defined for the Telegram deep link token expiration path — what does the user see in the web app if their token expires before they use it? Is auto-regeneration specified? [Gap, Spec §US-4] — **Resolved**: "Connect Telegram" always generates a fresh token; silently auto-regenerates. Added to spec.md Edge Cases and contracts/dashboard-api.md POST /settings/telegram-link.
- [x] CHK009 Are requirements specified for concurrent Telegram link attempts — what if a user generates multiple tokens before using any? [Gap, Spec §US-4 / data-model.md §telegram_link_tokens] — **Resolved**: Generating a new token deletes all previous tokens for the user. Added to data-model.md and contracts/dashboard-api.md.
- [x] CHK010 Are empty-state requirements defined for the dashboard (US-3) when a user has notes but gardening hasn't run yet? The spec covers "enough notes for clustering" but not the intermediate state. [Gap, Spec §US-3] — **Deferred**: Empty states explicitly deferred to hardening phase (spec Assumptions).

## Consistency

- [x] CHK011 Does the assumption "existing Cloudflare Pages vanilla JS/ES-modules SPA will be extended" (Spec §Assumptions) conflict with the plan's decision to use Next.js + Vercel in `webapp/`? The assumption needs updating or removal. [Conflict, Spec §Assumptions vs. plan.md §Project Structure] — **Resolved in F2**: spec.md Assumptions now states Next.js 14+ / Vercel / `webapp/`.
- [x] CHK012 Are the Dashboard API auth requirements consistent — dashboard-api contract says "static key retained as fallback during migration" but spec FR-009 and FR-016 imply all Workers use JWT-only user identity? Is the fallback in scope for MVP? [Conflict, Spec §FR-009 / contracts/dashboard-api.md §Authentication] — **Resolved in F4**: T059 removes the static fallback; it's a temporary deployment artifact only.

## Non-Functional Requirements

- [x] CHK013 Are capture and search latency budgets quantified? SC-005 mentions "without degradation" for 100 users but the plan mentions "capture < 5s, search < 2s" — are these in the spec's success criteria? [Gap, Spec §SC-005 vs. plan.md §Performance Goals] — **Accepted**: Latency numbers left as implementation guidance in plan.md only; SC-005 wording is sufficient for MVP.
- [x] CHK014 Are rate limiting requirements specified per-user for MVP, or is deferral to "hardening phase" an accepted risk? The assumption says deferred, but no FR gates the risk. [Assumption, Spec §Assumptions] — **Accepted**: Explicitly deferred in spec Assumptions.

## Dependencies & Assumptions

- [x] CHK015 Is the `mcp_api_key_hash` uniqueness constraint documented as a functional requirement? Data-model says "must be unique when set" but no FR covers the collision/rejection behavior if two keys somehow hash identically. [Gap, data-model.md §Validation Rules] — **Dismissed**: SHA-256 collision probability is negligible (~2^-128 per pair); not a meaningful FR.
- [x] CHK016 Are requirements for the `links` table user_id semantics specified — when two notes from the same user are linked, is the link's `user_id` always the same as both notes' `user_id`, and is cross-user linking explicitly prohibited? [Ambiguity, Spec §FR-004 / data-model.md §links] — **Pass**: Implicitly enforced by architecture — capture always runs in user context, so link.user_id = both notes' user_id; cross-user linking is architecturally impossible.
