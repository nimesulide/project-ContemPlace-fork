# Feature Specification: Multi-Tenant MVP

**Feature Branch**: `002-multi-tenant-mvp`
**Created**: 2026-03-29
**Status**: Draft
**Input**: User description: "ContemPlace Multi-Tenant MVP — row-level isolation (user_id + RLS), Supabase Auth, user identity threading across all Workers, per-user gardening, web application for capture/dashboard/settings, Telegram multi-user via DB lookup, data export"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - New User Signs Up and Captures First Note (Priority: P1)

A new user visits the ContemPlace web app, creates an account with email/password or a social login (Google/GitHub), and lands in an authenticated shell. They type a thought into the capture form, submit it, and see a structured note (title, body, tags) appear in their dashboard. Their data is completely isolated from all other users.

**Why this priority**: Without signup and capture, the product has zero value. This is the minimum viable path from "stranger" to "user with data."

**Independent Test**: Can be fully tested by signing up, submitting one capture, and verifying the structured note appears in the user's dashboard — delivers the core value proposition end-to-end.

**Acceptance Scenarios**:

1. **Given** the user has no account, **When** they sign up with email/password and verify their email, **Then** they are logged in and see the capture form.
2. **Given** the user has no account, **When** they sign up via Google OAuth, **Then** they are logged in and see the capture form without additional verification steps.
3. **Given** the user is logged in and has no notes, **When** they submit text in the capture form, **Then** the system returns a structured note with title, body, and tags, and the note appears in their dashboard.
4. **Given** two users exist, **When** user A captures a note, **Then** user B cannot see, search, or access that note through any interface.

---

### User Story 2 - User Connects MCP Client with Personal API Key (Priority: P1)

A signed-up user navigates to their settings page, sees their personal MCP endpoint URL and API key, copies both into their MCP-capable AI client (e.g., Claude.ai), and successfully uses `capture_note`, `search_notes`, and other MCP tools — all scoped to their own data.

**Why this priority**: The MCP endpoint is the core differentiator — "your knowledge accessible from any AI tool." Without per-user MCP, the product is just another notes app.

**Independent Test**: Can be tested by generating an API key in settings, configuring an MCP client, calling `capture_note` and `search_notes`, and verifying results are scoped to the user.

**Acceptance Scenarios**:

1. **Given** a logged-in user on the settings page, **When** they view their MCP configuration, **Then** they see a personal MCP endpoint URL and a masked API key with a copy button.
2. **Given** a user has an API key, **When** they configure an MCP client with their key and call `search_notes`, **Then** they see only their own notes.
3. **Given** a user has an API key, **When** they regenerate the key, **Then** the old key stops working immediately and the new key works.
4. **Given** a user provides an invalid or another user's API key, **When** they call any MCP tool, **Then** the request is rejected with an authentication error.

---

### User Story 3 - User Views Personal Dashboard with Clusters (Priority: P2)

A user with several captured notes visits their dashboard and sees their personal statistics, recent notes, and topic clusters — generated by the gardening pipeline running on only their data.

**Why this priority**: The dashboard makes the knowledge graph tangible. Without it, users can't see the value the system creates from their captures.

**Independent Test**: Can be tested by capturing 10+ notes over multiple topics, triggering gardening, and verifying the dashboard shows clusters and stats for only this user's notes.

**Acceptance Scenarios**:

1. **Given** a user has captured notes, **When** they visit the dashboard, **Then** they see statistics (note count, cluster count) based only on their own notes.
2. **Given** a user has enough notes for clustering, **When** the gardener has run on their data, **Then** the dashboard shows topic clusters with titles derived only from their notes.
3. **Given** two users each have notes, **When** each views their dashboard, **Then** each sees only their own clusters and stats — zero cross-user data leakage.

---

### User Story 4 - Existing User Connects Telegram for Mobile Capture (Priority: P2)

A signed-up user navigates to settings, clicks "Connect Telegram," receives a deep link, opens it in Telegram, sends a start command, and receives confirmation. From then on, messages sent to the bot are captured as notes in their account.

**Why this priority**: Telegram is the lowest-friction mobile capture channel. It retains existing single-tenant users and extends the product's capture surface.

**Independent Test**: Can be tested by connecting Telegram from settings, sending a message to the bot, and verifying the note appears in the user's web dashboard.

**Acceptance Scenarios**:

1. **Given** a logged-in user on settings, **When** they click "Connect Telegram," **Then** they see a unique deep link to the Telegram bot.
2. **Given** a user opens the deep link and sends the start command with token, **When** the bot processes it, **Then** the user's Telegram chat is linked to their account and they receive a confirmation message.
3. **Given** a user has connected Telegram, **When** they send a message to the bot, **Then** the message is captured as a note under their user ID and appears in their web dashboard.
4. **Given** a Telegram chat ID is not linked to any user, **When** a message arrives from that chat, **Then** the bot responds with instructions to connect via the web app.

---

### User Story 5 - User Exports All Their Data (Priority: P3)

A user navigates to settings and exports all their notes as a portable format, fulfilling the "your data is yours" promise.

**Why this priority**: Data portability is the trust contract. Without it, the "you own your data" claim is unsubstantiated. However, it doesn't block day-one usage.

**Independent Test**: Can be tested by capturing several notes, triggering export, and verifying the downloaded file contains all notes with titles, bodies, tags, raw input, and metadata.

**Acceptance Scenarios**:

1. **Given** a user has captured notes, **When** they request a data export, **Then** they receive a JSON file containing all their notes with titles, bodies, tags, raw input, creation dates, and link metadata.
2. **Given** a user has no notes, **When** they request a data export, **Then** they receive a valid JSON file with an empty notes array.

---

### User Story 6 - Automated Provisioning on Signup (Priority: P1)

When a new user completes signup, the system automatically creates their profile, seeds a default capture voice, and generates an MCP API key — without any manual intervention or admin action.

**Why this priority**: Zero-friction onboarding is a prerequisite for self-service. If provisioning fails silently, the user gets a broken experience on first use.

**Independent Test**: Can be tested by creating a new account and immediately verifying that the user profile, capture voice, and API key all exist in the database.

**Acceptance Scenarios**:

1. **Given** a new user completes signup, **When** the provisioning process runs, **Then** a user profile row, a default capture voice profile, and a hashed MCP API key all exist for that user.
2. **Given** a new user completes signup, **When** they navigate to settings and click "Generate API Key," **Then** a key is created, displayed once, and ready to use immediately.
3. **Given** the provisioning trigger encounters an error, **When** the user tries to capture a note, **Then** the system provides a clear error message rather than failing silently.

---

### Edge Cases

- What happens when a user deletes their account? All their data (notes, links, clusters, profiles, connections) must be cascade-deleted. Self-serve account deletion UI is deferred to post-MVP; cascade delete is enforced at the DB level via FK ON DELETE CASCADE.
- What happens when the same Telegram chat ID attempts to connect to two different user accounts? The system must reject the second connection and inform the user the chat is already linked.
- What happens when a user clicks "Connect Telegram" and already has a pending or expired token? The system always generates a fresh token and deletes all previous tokens for that user — the user sees a new valid deep link each time they click.
- What happens when the gardener encounters a user with zero notes? It skips that user gracefully without errors.
- What happens when the provisioning trigger fails mid-execution (e.g., capture voice seeded but API key generation fails)? The system must be idempotent — retrying provisioning completes the missing steps without duplicating existing ones.
- What happens when a user's session expires while they're mid-capture? The capture form preserves the user's input and prompts re-authentication, then completes the submission.
- What happens when the existing single-tenant data is migrated? All existing notes, links, clusters, and enrichment data must be attributed to the original owner with no data loss.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST support user registration via email/password with email verification.
- **FR-002**: System MUST support user registration via Google and GitHub OAuth.
- **FR-003**: System MUST automatically provision a user profile, default capture voice, and MCP API key upon account creation.
- **FR-004**: System MUST isolate all user data at the row level using a user identifier on every data table (notes, links, clusters, enrichment_log, capture_profiles, processed_updates, entity_dictionary).
- **FR-005**: System MUST enforce that no user can read, write, search, or access another user's data through any interface (web, MCP, Telegram, dashboard).
- **FR-006**: System MUST provide a web-based capture form that accepts text input and returns structured notes (title, body, tags) via the existing capture pipeline.
- **FR-007**: System MUST provide each user a personal MCP endpoint authenticated by a per-user API key.
- **FR-008**: System MUST allow users to view and regenerate their MCP API key from a settings page.
- **FR-009**: System MUST thread user identity from authentication through every database operation in all Workers (MCP, Telegram, Gardener, Dashboard API).
- **FR-010**: System MUST run the gardening pipeline (similarity linking, clustering, entity extraction) per-user, isolated from other users' data.
- **FR-011**: System MUST allow users to connect their Telegram account via a one-time deep link token flow initiated from the web app settings.
- **FR-012**: System MUST resolve incoming Telegram messages to a user identity via database lookup, replacing the static chat ID whitelist.
- **FR-013**: System MUST provide a user-scoped dashboard showing statistics, recent notes, and topic clusters for only the authenticated user's data.
- **FR-014**: System MUST provide a data export capability that returns all of a user's notes and metadata as JSON.
- **FR-015**: System MUST update all database search and query functions to accept and filter by user identity.
- **FR-016**: System MUST apply row-level security policies as defense-in-depth, ensuring that even service-role queries explicitly filter by user identity.
- **FR-017**: System MUST migrate all existing single-tenant data to a designated owner user during the multi-tenant migration.
- **FR-018**: System MUST preserve the existing capture pipeline behavior — the multi-tenant change adds user scoping without altering capture quality, linking, or voice correction.

### Key Entities

- **User Profile**: Represents a registered user. Attributes: user identity (from auth provider), hashed MCP API key, subscription plan, creation date.
- **Capture Voice Profile**: Per-user configuration for the capture LLM's tone, style, and body rules. One per user, seeded at signup with system defaults.
- **Telegram Connection**: Links a Telegram chat to a user account. Attributes: user identity, Telegram chat ID, connection timestamp. One chat ID maps to exactly one user.
- **MCP API Key**: Per-user bearer token for MCP endpoint authentication. Stored as a hash. Regeneratable by the user; old key invalidated immediately on regeneration.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A new user can go from landing page to first captured note in under 3 minutes.
- **SC-002**: Zero cross-user data leakage — no user can access another user's notes, clusters, or metadata through any system interface.
- **SC-003**: A user can configure their MCP client and successfully capture/search notes within 2 minutes of copying their API key.
- **SC-004**: The gardening pipeline processes each user's data independently — a failure in one user's gardening run does not affect any other user.
- **SC-005**: The system supports at least 100 concurrent registered users without degradation in capture or search response times.
- **SC-006**: All existing single-tenant data is preserved and correctly attributed to the original owner after migration.
- **SC-007**: A user can export all their data as a portable file at any time.
- **SC-008**: Telegram connection flow completes in under 1 minute from clicking "Connect" to receiving confirmation in the bot.

## Assumptions

- Row-level isolation (user_id column + RLS policies) is the chosen tenant isolation strategy. Schema-per-tenant is a future upgrade path if needed.
- Supabase Auth is the identity provider — it is already available in the existing Supabase project and requires no new infrastructure deployment.
- The web app is a new Next.js 14+ (App Router) project in `webapp/`, deployed to Vercel. It replaces the role of the existing `dashboard/` vanilla JS SPA. Styling is exclusively MUI v5 theming and `sx` props.
- Workers continue to use the service_role key to bypass RLS, but always explicitly filter by user_id in application code. RLS serves as defense-in-depth, not primary access control.
- OAuth providers for MVP are limited to Google and GitHub. Additional providers can be added post-MVP without architectural changes.
- Sequential per-user gardener iteration is sufficient for MVP scale (up to ~100 users). Queue-based processing is deferred.
- The web capture form uses `source: 'web'` to distinguish from Telegram captures.
- Existing data belongs to a single "founder" user who will be created/migrated as the first real auth user.
- Rate limiting, usage tracking, and detailed onboarding UX (empty states, guided first capture) are deferred to the hardening phase after core multi-tenancy ships.
- Self-serve account deletion UI is deferred to post-MVP. Data integrity on deletion is enforced by DB-level CASCADE constraints.
- OAuth identity linking (same email registered via both email/password and OAuth provider) relies on Supabase Auth default behavior for MVP. Explicit cross-provider account merging is deferred.
- Data export format is JSON. Additional formats (CSV, Markdown) are deferred.
